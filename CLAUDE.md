# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project intent

This repo is a collection of Roslyn **source generators** for Unity, plus a Unity project that consumes and tests them. Authoritative intent lives in `docs/Directives.md` — read it before making structural decisions.

State as of this writing: `src/generators-src/` (the generators) is empty. The Unity side (`src/generators-unity/`) is a fresh Unity 6 project (6000.4.0f1, URP) with only Unity's default `Readme` sample scripts. Almost any first work here is greenfield — there is no existing generator code to mirror.

## Repository layout

- `src/generators-src/` — Roslyn source generator projects (.NET / netstandard2.0 expected by Unity). Currently empty; create the generator csproj(s) here.
- `src/generators-unity/` — Unity 6 project used to test/consume the generators. Open this folder in the Unity Editor. The generator DLLs (once built) get dropped into `Assets/` with the appropriate Roslyn analyzer asset labels so Unity picks them up.
- `docs/Directives.md` — project-level rules. The two non-obvious ones to remember:
  - **Templates ship as embedded text resources** inside the generator dll (not string literals in code).
  - **Placeholders in those templates use `%%Name%%`** (double percent), e.g. `%%ClassName%%`.
  - When something a generator needs looks reusable, extract it into a shared `Helpers` class **immediately** rather than waiting for the second generator to need it.
- `docs/Coding Conventions.md` — naming/formatting rules that apply to **both** the generator code and the Unity-side C#. Highlights that bite if you forget:
  - Allman braces, always present (even for one-liners).
  - Private fields `_camelCase`; serialized private fields use `[SerializeField] private`; **no public fields ever** — expose serialized values via `[field: SerializeField]` properties, never via auto-properties.
  - File member order: constants → static → events → properties → fields → readonly → ctors → public methods → private methods → Dispose/OnDestroy → destructor → nested types. Within each, public → protected → private.
  - Flags enums: plural name, `1 << n` values, `0 = None`. Non-flags enums: singular, reserve `0` for None/Empty.
  - No `#region`. No abbreviations (except math). `nameof()` for member references; interpolation over concatenation.

## Build & run

There is no top-level build script. Two independent toolchains:

- **Generators (`src/generators-src/`):** standard .NET. Once a csproj exists, build with `dotnet build src/generators-src/<Project>.csproj -c Release`. Source generators must target `netstandard2.0` and reference `Microsoft.CodeAnalysis.CSharp` for Unity compatibility. Output dll is copied (or symlinked) into the Unity project's `Assets/` and configured as a Roslyn analyzer in the asset's import settings (`RoslynAnalyzer` label, no platforms checked).
- **Unity project (`src/generators-unity/`):** open in Unity 6000.4.0f1 (see `ProjectSettings/ProjectVersion.txt` — match the version or Unity will upgrade the project). The `.sln` / `.csproj` files in this folder are **auto-generated by Unity** — don't hand-edit them.

## Testing

Unity has the Test Framework package installed (`com.unity.test-framework` 1.6.0). Per `Directives.md`, tests for the generators live on the Unity side. Run via **Window → General → Test Runner** in the Editor, or headless:

```
/Applications/Unity/Hub/Editor/6000.4.0f1/Unity.app/Contents/MacOS/Unity \
  -batchmode -nographics -projectPath src/generators-unity \
  -runTests -testPlatform EditMode -logFile -
```

No test assemblies exist yet; the first generator work should add an `*.asmdef` with `"Test Assemblies"` references under `Assets/Tests/` (or similar) before adding tests.

## Conventions worth re-stating

- One MonoBehaviour per file; MonoBehaviour class name **must** match the filename.
- Namespaces mirror assembly namespace + folder structure.
- Don't reach for `var` when the right-hand side doesn't make the type obvious.
- Events: present participle = "about to happen" (`OpeningDoor`), past participle = "just happened" (`DoorOpened`); handlers prefix with `On`.
